# 港口环境实例信息文件模板

`<WIKI_BASE>/<域>/_env/<港口码>.md` 的固定骨架。占位符 `<...>` 全部替换后落盘；
校验器 `scripts/validate_env.py` 强制以下结构，改结构先改校验器再改模板。

## 结构规则（校验器强制）

1. frontmatter 必须含 `port` / `domain` / `updated` 三键；
2. 二级节只允许 `## test` / `## Pre-release` / `## prod` / `## 待确认 / 疑问`，三个环境节缺一不可（该港口没有此环境时正文写「无」，节不能省）；
3. 实例 = 三级节，标题必须 `### <环境> · <实例名>`（前缀与所在环境节一致），全文件唯一——标题即锚点，速查根因叶 / 子系统 §8 都靠它链接，改名前先跑 `--check-refs`；
4. MySQL、Redis、各服务、中间件都是「实例」，不单列层级；
5. 实例正文自由行文，推荐信息行：位置 / 日志 / 配置文件 / 脚本·工具 / 凭据获取；JumpServer、Grafana 等取证入口并入相关实例的正文，不另立节；
6. **凭据获取只写指针**（哪个脚本 / 哪台机器哪个文件 / 哪个 Mcp 配置叶子），密码/token 明文触发校验器黑名单直接拒绝；
7. 每个实例节带 `> 来源：<用户口述|xx skill 摘录>（YYYY-MM-DD）`——环境信息会漂移，日期即时效标记；
8. 不确定的事实不进环境节正文，进 `## 待确认 / 疑问`，条目带「已检查 / 建议核实方向」。

## 模板

```markdown
---
port: <港口码>
domain: <域>
title: <港口码>（<港口中文名>）环境实例信息
updated: <YYYY-MM-DD>
---

# <港口码>（<港口中文名>）环境实例信息

> 本文档由 env-quickref skill 维护，只存「哪里有什么」（实例位置、日志/配置/脚本入口）；
> 「怎么取证」见各工具 skill（fms-log / fms-diagnose / cmdcap）。凭据一律只写指针。

## test

### test · <实例名>

- 位置：<host:port / 集群 / namespace>（未知写 待补）
- 日志：<路径 / datasource / fms-log --port 参数>
- 配置文件：<nacos 地址·namespace / 文件路径>
- 脚本·工具：<相关脚本、取证工具及位置>
- 凭据获取：<指针>

> 来源：<用户口述>（<YYYY-MM-DD>）

## Pre-release

无

## prod

无

## 待确认 / 疑问

- [<环境> · <实例名>] <待核实内容>（已检查 <已排除项>；建议核实方向 <怎么确认>）
```
