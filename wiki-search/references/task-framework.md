# 任务处理框架（Phase 1–5 详解）

本文是五阶段任务处理框架的详解：任务拆解 → Wiki 检索 → 排查工具证据收集 → 问题定位 → 改进建议与风险清单。每阶段给出执行要点与「输出物」规格，文末以一个故障排查任务完整走通五阶段。全程只读：绝不修改 wiki 下任何文件。

## 0. 路径占位符

本文命令与路径一律用占位符书写，替换为本机解析值即可运行：

| 占位符 | 含义 | 解析方法 |
|---|---|---|
| `<WIKI_BASE>` | wiki 根目录 | 读用户主目录 `~/.document-systems.json` 的 `wiki_base` 字段 |
| `<DOMAIN>` | 仓所属业务域 | wiki_engine `resolve-domain` 解析 |
| `<DOC_ROOT>` | 目标仓的文档根目录 | `<WIKI_BASE>\<DOMAIN>\<仓名>` |
| `<子系统>` | 仓内子系统目录名 | `<DOC_ROOT>` 下一级子目录，各含一份 `architecture.md` |

解析 `<WIKI_BASE>`（PowerShell）：

```powershell
(Get-Content "$env:USERPROFILE\.document-systems.json" -Encoding UTF8 | ConvertFrom-Json).wiki_base
```

`<DOMAIN>` / `<DOC_ROOT>` 的完整解析步骤与 Git Bash 版命令，见 [search-strategy.md](./search-strategy.md) §1。

## 1. 框架总览

| 阶段 | 职责（一句话） | 输出物 |
|---|---|---|
| Phase 1 任务拆解 | 回答三问，先理解后行动 | 三问的明确答案（检索关键词 + 收敛目标） |
| Phase 2 Wiki 检索 | 按场景顺序查 wiki，收集「应然」 | 原文片段引用：「在 `<文件路径>` §N 中找到：…」 |
| Phase 3 排查工具证据收集 | 按任务类型调排查工具，收集「实然」 | 运行时证据（标注工具名 + 参数） |
| Phase 4 问题定位 | 应然 vs 实然的 Gap → 收敛到唯一根源 | 根源陈述（主体 + 位置 + 根因事实，一句话） |
| Phase 5 改进建议与风险清单 | 给出可执行方案并全面评估风险 | 4 行建议 + R1–R8 风险清单 |

五阶段依序执行。非故障类任务（理解模块、实现前调研、改动评估）同样走此框架，只是 Phase 2 走普通分支，Phase 4 的收敛对象是任务问题的答案而非故障根源。

## 2. Phase 1 — 任务拆解（三问）

进入框架后第一件事：回答三问。三问答案决定 Phase 2 检索什么（关键词、走哪条分支）和 Phase 4 收敛到什么（目标）。

### 2.1 三问清单

| # | 问题 | 要得到的答案 | 示例 |
|---|---|---|---|
| Q1 | **What（是什么）** | 主体对象：API / 模块 / 业务流程 / 故障现象——名字是什么、属于哪个仓/子系统 | `fms-gateway 鉴权`（API）、`JobOrder 卡死`（故障现象） |
| Q2 | **Why（为什么）** | 任务背景：生产故障 / 新功能 / 性能优化 / 技术债 | 生产问题、需求评审 |
| Q3 | **What（要实现什么）** | 功能边界：具体改什么/查什么，边界在哪（哪些不做） | 分析根因 / 实现接口 / 评估改动 |

### 2.2 每问的提问模板

自问模板（从任务描述中抽取答案）：

- **Q1**：「本任务的主体对象是哪一个 API / 模块 / 业务流程 / 故障现象？名字是什么？属于哪个仓、哪个子系统？」
  示例答案：「主体 = fms-gateway 的鉴权接口（API）」「主体 = 车辆离线（故障现象），属仓 `fms-server`（示例值）」
- **Q2**：「任务因何而起——生产故障、新功能、性能优化、还是技术债？」
  示例答案：「生产故障：现场作业受影响」「新功能：需求评审要求新增接口」
  （Q2 直接决定 Phase 2 走故障分支还是普通分支，见 §3.1。）
- **Q3**：「要交付什么：改什么 / 查什么？做到哪一步为止——只定位、给方案、还是动代码？哪些明确不做？」
  示例答案：「只定位根因并给建议，不动代码」「实现 X 接口，不含前端改造」

判定「答清了没有」：每问都能用一句话填出唯一答案 = 清楚；填不出，或存在多个互斥候选 = 不清楚。

### 2.3 答不清就先确认（规则）

**答不清哪一问，就先向用户确认哪一问**；确认清楚后才进入 Phase 2。一次只问缺的那几问，已明确的不重复问。向用户确认的提问模板：

- Q1 不清：「这次要处理的对象具体是哪一个（API / 模块 / 流程 / 现象）？比如是 A 还是 B？」
- Q2 不清：「任务背景是生产故障、新功能、性能优化还是技术债？——这决定先查诊断树还是直接查架构文档。」
- Q3 不清：「交付边界到哪：只定位原因，还是给出改动方案，还是直接改代码？」

> **输出物**：三问的明确答案（各一句话）。Q1/Q3 提供 Phase 2 的检索关键词与 Phase 4 的收敛目标，Q2 决定 Phase 2 的执行分支。

## 3. Phase 2 — Wiki 检索（执行顺序）

### 3.1 场景判定

| 判定条件 | 走哪条路径 |
|---|---|
| Q2 = 生产故障，或任务里有「要解释的异常现象」（排查 / 定位 / 报错 / 告警 / 变慢 / 为什么 X 不工作） | **§3.2 故障定位场景**：先诊断树，再架构文档 |
| 其余（新功能 / 无故障现象的性能优化 / 技术债 / 理解模块 / 评估改动） | **§3.3 普通任务场景**：直接分层检索 |

一句话判据：**有现象要解释 → 先查诊断树；只有事情要做/要了解 → 直接分层检索。**

### 3.2 故障定位场景：先诊断树，再架构文档

1. **先读该仓诊断树** `<DOC_ROOT>\生产问题速查.md`：

   ```powershell
   Get-Content "<DOC_ROOT>\生产问题速查.md" -Encoding UTF8
   ```

   按「现象根（`##`）→ 判别分支 → 根因叶」走树（树的详细用法见 [search-strategy.md](./search-strategy.md) §2）：
   - **命中**：直接引用原文片段（含路径与现象根标题），叶子上的候选根因与「确认 →」锚点作为 Phase 3 的取证起点。**速查命中 ≠ 定案**——候选根因必须经 Phase 3 证据确认。
   - **未命中**：如实注明「速查暂无此现象」；文件不存在则注明「该仓暂无生产问题速查」。两种情况都继续第 2 步。
2. **再查架构文档补上下文**：走 §3.3 的分层检索；故障场景优先读 §9 已知问题、§6 关键流程。

### 3.3 普通任务场景：直接分层检索

1. 解析 `<DOC_ROOT>`（§0）；
2. 分层 grep：仓（`<DOC_ROOT>`）→ 域（`<WIKI_BASE>\<DOMAIN>`）→ 全 wiki（`<WIKI_BASE>`），上一层 0 命中才升层；
3. 按任务类型节定向：接口→§4、依赖→§5、流程→§6、数据→§7、配置→§8、已知问题→§9（完整映射表与可复制命令见 [search-strategy.md](./search-strategy.md) §4–§5）。

### 3.4 输出物规格（两场景通用）

> **输出物**：wiki 原文片段引用，格式固定为——
> - 架构文档：「在 `<文件路径>` §N 中找到：<原文片段>」
> - 诊断树：「在 `<DOC_ROOT>\生产问题速查.md`「<现象根标题>」中找到：<原文片段>」
>
> **直接引用原文片段，不转述。**

正 / 反例：

- ✅ 在 `<DOC_ROOT>\<子系统>\architecture.md` §6 中找到：「车辆心跳每 3s 写入 Redis Stream，在线判定 = 最近 10s 内有心跳」（示例值，虚构片段）
- ❌ 「架构文档里说心跳走 Redis、超时就算离线」——转述，且无路径无节号、无法核查，禁止。

## 4. Phase 3 — 排查工具证据收集

wiki 提供「设计意图」（应然），排查工具提供「运行时实况」（实然），两类信息缺一不可。按任务类型选工具（非互斥，可并用）：

| 任务类型 | 工具 |
|---|---|
| FMS 业务故障（车辆离线 / 任务失败 / 调度异常 / 告警…） | fms-diagnose |
| 查具体错误日志、接口 500/慢、异常堆栈 | fms-log |
| 需在堡垒机后的远程服务器执行命令取结果 | cmdcap |
| 代码调用链追踪 / 陌生模块执行流 / 改动影响面 | gitnexus-debugging / gitnexus-exploring / gitnexus-impact-analysis |

选型细则、各工具的命令用法与输出解读见 [diagnostic-tools.md](./diagnostic-tools.md)。

> **输出物**：运行时证据（诊断结论 / 日志片段 / 命令输出 / 调用链图），每条证据标注来源（工具名 + 参数），保证可复现。

## 5. Phase 4 — 问题定位（收敛到唯一根源）

综合 Phase 2（应然）与 Phase 3（实然）定位问题。

### 5.1 收敛判定准则

- ✅ **已收敛**：能写出「根源在 `<具体文件/类/配置>` 的 `<具体位置>`，原因是 `<具体事实>`」——三个空位都填了具体值，且每个断言有 Phase 2/3 的证据支撑。
- ❌ **未收敛**：「可能是 A，也可能是 B」——存在两个以上未被排除的候选。**必须继续收集证据**（回 Phase 2 补检索 / 回 Phase 3 补取证），把候选排除到唯一，才能进入 Phase 5。

### 5.2 收敛过程（Gap 法）

1. Phase 2 的 wiki 片段确认**应然**：设计上系统应该怎样工作；
2. Phase 3 的运行时证据确认**实然**：当前实际是什么状态；
3. **应然与实然的差异（Gap）= 根源**。找不到 Gap 时，说明主体对象找错（回 Phase 1）或证据不足（回 Phase 2/3）；不允许在没有 Gap 支撑时硬写结论。

### 5.3 正 / 反例

| 陈述（示例值） | 判定 | 依据 / 下一步 |
|---|---|---|
| 「根源在 `fms-vehicle-bridge` 的车端会话管理（`bridge:vehicles` 会话注册），原因是会话过期后未自动重建，心跳中断超过在线判定窗口」 | ✅ 已收敛 | 主体、位置、事实齐全且有证据 → 进 Phase 5 |
| 「根源在 `<子系统>` 的网关重试配置（`retry.max-attempts`），原因是对 5xx GET 自动重试，把一次前端轮询放大成 4 倍请求」 | ✅ 已收敛 | 定位到具体配置项 + 机制事实 → 进 Phase 5 |
| 「可能是车端网络问题，也可能是 Redis 挂了」 | ❌ 未收敛 | 两个互斥候选并存 → 继续取证排除（如先用 fms-diagnose 的 Redis / 流积压类检查排除一侧） |
| 「大概率是心跳超时导致的」 | ❌ 未收敛 | 只有方向，没有位置（哪个类/配置）也没有事实证据 → 回 Phase 2 节定向 §6 找心跳机制的具体组件，再取证 |
| 「重启后恢复了，应该没问题了」 | ❌ 未收敛 | 现象消失 ≠ 根源定位，写不出根源陈述 → 仍须走完取证与收敛 |

> **输出物**：**根源陈述** = 主体对象 + 位置 + 根因事实，一句话；代码标识符（类名 / 配置键 / 表名 / 错误类）原样保留，不意译。

## 6. Phase 5 — 改进建议与风险清单

### 6.1 改进建议（4 行格式）

```text
现象：<用户感知到的问题/需求>
根源：<Phase 4 的根源陈述>
方案：<具体改动内容，精确到文件/方法/配置>
参考：[<wiki 文档 §节>](<相对路径#锚点>)
```

各行要求：

- **现象**：用户视角的表述（用户能对上号），不堆内部术语；
- **根源**：原样复用 Phase 4 的根源陈述，不另写一版；
- **方案**：可直接执行——精确到文件 / 方法 / 配置项，不写「优化一下 X」这类空话；
- **参考**：指向支撑本建议的 wiki 文档 §节，写**相对 `<DOC_ROOT>` 的路径 + `#` 锚点**；锚点按节标题的 markdown slug 规则生成（去掉 `.` 等标点、空格换 `-`、英文转小写）。示例（示例值）：`[fms-vehicle-bridge §6 关键流程](./fms-vehicle-bridge/architecture.md#6-关键流程)`。多条参考可并列多个链接。

### 6.2 风险清单

对 R1–R8 八个方向（兼容性 / 数据 / 性能 / 依赖 / 部署 / 回滚 / 测试覆盖 / 已知问题叠加）逐项标注「高 / 中 / 低 / 不适用」，**高风险项必须给出缓解措施**。方向定义、检查点与辅助工具见 [risk-checklist.md](./risk-checklist.md)。

> **输出物**：4 行改进建议 + R1–R8 风险清单（高风险项带缓解措施）。

## 7. 端到端示例：NP FMS 车辆离线排查

> 本节演示五阶段的完整走法与各阶段输出物格式。示例中的域名 / 仓名 / 子系统名 / 组件名、引用的文档片段与工具输出**均为示例值（虚构示意）**；实际执行以本机占位符解析结果与真实文档内容为准。

**任务输入**：「NP 港有一辆集卡长时间离线，任务一直分配不出去，帮我排查。」

### Phase 1 — 三问

- **What（是什么）**：故障现象 =「车辆离线 + 任务分配不出去」；属 NP FMS，仓 = `fms-server`（示例值）。
- **Why（为什么）**：生产故障（现场作业受影响）→ Phase 2 走故障定位分支（§3.2）。
- **What（要实现什么）**：定位根因，给出改进建议与风险评估；不改代码、不重启服务。

三问均能一句话作答 → 无需确认，直接进 Phase 2。（对照 §2.3：若用户没说是哪个港，Q1 答不清，应先问「是 NP 还是 PSA？两个港口架构不同」。）

### Phase 2 — 先查诊断树，再查架构文档

解析占位符（方法见 §0，解析结果因机器而异）。本例解析结果（示例值）：`<DOC_ROOT>` = `D:\wiki\NP_FMS\fms-server`。

**① 读诊断树：**

```powershell
Get-Content "<DOC_ROOT>\生产问题速查.md" -Encoding UTF8
```

命中时的输出物写法：

> 在 `<DOC_ROOT>\生产问题速查.md`「车辆离线 / 任务分配不出去」中找到（示例值，虚构片段）：
> 「**判别**：单车离线、其余车正常；Redis 在线集合 `vehicle:online:name` 无该车 → **根因**：fms-vehicle-bridge 与车端会话断开后未重建，心跳未再进入在线集合。**确认** → fms-vehicle-bridge/architecture.md §6」

→ 得到候选根因（会话断开未重建）与确认锚点（§6），作为 Phase 3 的取证起点。

**未命中时的输出物写法**（规范，分两种情况）：

> - 现象不在树上：「已读 `<DOC_ROOT>\生产问题速查.md`，**速查暂无此现象**（现有现象根：…，均不匹配）→ 转架构文档分层检索。」
> - 文件不存在：「**该仓暂无生产问题速查** → 直接转架构文档分层检索。」
>
> 转入分层检索后，故障场景优先读 §9 已知问题、§6 关键流程（§3.2 第 2 步）。

**② 架构文档补上下文**（沿「确认 →」锚点节定向）：

```powershell
Get-Content "<DOC_ROOT>\fms-vehicle-bridge\architecture.md" -Encoding UTF8 |
  Select-String -Pattern "^## 6\." -Context 0,50
```

输出物：

> 在 `<DOC_ROOT>\fms-vehicle-bridge\architecture.md` §6 中找到（示例值，虚构片段）：
> 「车辆经 bridge 建立会话并注册到 `bridge:vehicles`；心跳每 3s 写入 Redis Stream `vehicle:reports`；在线判定 = 最近 10s 内有心跳（维护于集合 `vehicle:online:name`）；不在线的车不参与派单。」

至此拿到「应然」：会话注册 + 心跳入流 + 10s 在线窗口 + 在线才派单。

### Phase 3 — fms-diagnose / fms-log 取证

按 §4 选型：FMS 业务故障 → fms-diagnose；日志佐证 → fms-log。命令中 `<SKILLS_ROOT>` 为 skill 安装根目录（按「项目级 `.claude/skills/` → 用户级 `~/.claude/skills/`」顺序探测，详见 [diagnostic-tools.md](./diagnostic-tools.md) §1.1）：

```bash
python -X utf8 "<SKILLS_ROOT>/fms-diagnose/scripts/fms_diagnose.py" --list                              # 先列出可用检查项
python -X utf8 "<SKILLS_ROOT>/fms-diagnose/scripts/fms_diagnose.py" --port np --check vehicle-online    # 在线集合 vs 注册车队
python -X utf8 "<SKILLS_ROOT>/fms-diagnose/scripts/fms_diagnose.py" --port np --check np-bridge-health  # NP 专属：bridge 会话/心跳
```

返回（示例值）：

- `vehicle-online`：`status=warn`，`headline="注册车队中 1 辆不在在线集合：AT-07"`。【来源：fms-diagnose `--port np --check vehicle-online`】
- `np-bridge-health`：`status=alert`，`headline="bridge:vehicles 中无 AT-07 会话；route:registry 心跳已过期"`。【来源：fms-diagnose `--port np --check np-bridge-health`】

```bash
python -X utf8 "<SKILLS_ROOT>/fms-log/scripts/fms_log.py" --port np --module fms-vehicle-bridge --grep "ERROR|session" --ignore-case --context 2
```

返回（示例值）：`session expired for vehicle AT-07, closing stream consumer`，时间点与车辆掉线时刻吻合。【来源：fms-log `--port np --module fms-vehicle-bridge --grep "ERROR|session"`】

### Phase 4 — 收敛

- **应然**（Phase 2）：车辆会话注册于 `bridge:vehicles`，心跳每 3s 入流，最近 10s 有心跳才算在线，在线才派单；
- **实然**（Phase 3）：AT-07 会话过期被关闭、`bridge:vehicles` 无其会话、在线集合无该车，日志显示会话过期后消费者关闭；
- **Gap**：设计上会话应持续维持心跳链路，实际会话过期后链路中断且未重建。

**根源陈述**（输出物）：

> 根源在 `fms-vehicle-bridge` 的车端会话管理（`bridge:vehicles` 会话注册与心跳链路，示例值），原因是 AT-07 的会话过期后未自动重建，心跳中断超过 10s 在线判定窗口，车辆被判离线、不再参与派单。

收敛自检：单一根源 ✅；位置具体（到组件/机制）✅；每个事实有证据 ✅。——若此时仍存在「也可能是车端网络断了」的候选，则按 §5.1 判为未收敛，需补充证据（如查流积压、车端侧日志）排除后才能出此陈述。
