# 改动风险清单（R1–R8）

配合改进建议使用（五阶段框架 Phase 5）：在给出改动方案后、实施前，对照 R1–R8 八个风险方向逐项检查、标注等级；**高风险项必须给出缓解措施建议**。目的是让风险评估全面、不遗漏，而不是把每项都写成高风险。

## 使用方法

1. **逐方向检查**：R1–R8 每个方向都要过一遍全部检查点，不允许跳过；与本次改动无交集的方向标「不适用」并写明理由。
2. **用辅助工具取证**：wiki §节提供设计意图（接口契约、依赖、已知问题），gitnexus 系列与 fms-diagnose / fms-log 提供代码与运行时实况。检查结论要有出处（文档路径 + §节，或工具名 + 参数），不凭记忆下判断。
3. **按标注规则定级**：见下文「标注规则」。
4. **产出**：复制「空白模板」填写，随改进建议一并输出。

### 辅助工具与路径说明

- `wiki §N` 指目标子系统 `architecture.md` 的第 N 章，路径 `<DOC_ROOT>\<子系统>\architecture.md`（单子系统仓库为 `<DOC_ROOT>\architecture.md`）。
- 占位符解析：`<WIKI_BASE>` = 用户主目录 `~/.document-systems.json` 中 `wiki_base` 字段的值；`<DOC_ROOT>` = `<WIKI_BASE>\<DOMAIN>\<仓名>`，其中 DOMAIN 用 document-systems 的 wiki_engine 解析：

  ```powershell
  python -X utf8 <ENGINE_CLI> resolve-domain --repo <REPO_ROOT> --wiki <WIKI_BASE>
  ```

  解析结果示例（本行均为示例值，非固定路径）：若 `wiki_base` 为 `D:\wiki`、DOMAIN 解析为 `NP_FMS`、仓名为 `fms-server`，则 `<DOC_ROOT>` = `D:\wiki\NP_FMS\fms-server`。

- 读取某一 §节（PowerShell，注意 `-Encoding UTF8`）：

  ```powershell
  Get-Content "<DOC_ROOT>\<子系统>\architecture.md" -Encoding UTF8 |
    Select-String -Pattern "^## 4\." -Context 0,50
  ```

- 检索策略详见 `search-strategy.md`；fms-diagnose / fms-log / gitnexus 系列的调用方法详见 `diagnostic-tools.md`。

## R1–R8 风险方向与检查点

| # | 风险方向 | 检查点 | 辅助工具 |
|---|---------|--------|---------|
| R1 | **兼容性风险** | ① 改动是否影响对外接口（§4）？② 已知客户端是否依赖此接口？③ API 版本策略？ | wiki §4 |
| R2 | **数据风险** | ① 是否影响 DB 表结构（§7）？② 是否需要迁移脚本？③ 现有数据是否仍合法？ | wiki §7 + fms-diagnose |
| R3 | **性能风险** | ① 改动是否在热点流程（§6）上？② 是否引入额外 DB/Redis/网络调用？③ 吞吐/延迟影响？ | wiki §6 + fms-log |
| R4 | **依赖风险** | ① 改动是共享库还是独立服务？② 所有消费方是否受影响（§5 下游）？③ 是否需要多仓联动？ | wiki §5 + gitnexus-impact-analysis |
| R5 | **部署风险** | ① 多服务上线顺序？② 配置变更（§8）是否同步？③ 灰度策略可行？ | wiki §8 |
| R6 | **回滚风险** | ① 改动是否可逆？② 回滚步骤？③ 回滚窗口期？ | — |
| R7 | **测试覆盖风险** | ① 改动路径有无现有测试？② 是否需要补测试再上线？③ 边界条件覆盖？ | gitnexus-debugging |
| R8 | **已知问题叠加** | ① §9（已知问题）是否有与本次改动交叉的条目？② 改动是否可能触发 §9 缺陷？③ 多个已知问题是否可能与本次改动叠加放大（组合风险）？ | wiki §9 |

检查点中的 §N 均指目标子系统 `architecture.md` 的对应章节；「—」表示该方向无固定辅助工具，靠方案本身推演。

## 标注规则

每个方向标注为 **高 / 中 / 低 / 不适用** 之一：

| 标注 | 判定标准 | 要求 |
|------|---------|------|
| **高** | 检查点命中，且满足任一情形：改动影响对外契约且已有消费方依赖旧行为；波及面跨模块/跨仓、需联动变更或联动发版；出错后果严重（数据丢失、不可逆、生产中断）且无既有防护 | **必须给出缓解措施建议** |
| **中** | 检查点命中，但影响可控：有兼容层、可灰度、数据可重建、波及面限于单模块等 | 建议给出缓解措施 |
| **低** | 检查点基本不命中，或影响极小、常规发布流程即可覆盖 | 写一句判定依据 |
| **不适用** | 该方向与本次改动无交集（例：纯文档改动之于 R2 数据风险） | 写明不适用理由，不允许留空 |

补充原则：

- **拿不准时不降级**：证据不足以判「低」时，先用辅助工具补充证据；仍无法判定的，按更高一级标注。
- **高风险项必须给出缓解措施建议**。缓解措施要具体到操作层面（双写兼容、迁移脚本、联动发版顺序、新增监控项等），只写「注意风险」不算缓解措施。

## 填写示例

> 场景（示例）：修改 fms-core 共享库 `JobOrderCacheManager` 的 Redis key 格式。以下检查结论、路径与引用均为示例值，用于演示填写方法。

### 改动风险清单：修改 fms-core 共享库 JobOrderCacheManager 的 Redis key 格式

| # | 风险方向 | 辅助工具 | 检查结论 | 标注 | 缓解措施 |
|---|---------|---------|---------|------|---------|
| R1 | 兼容性风险 | wiki §4 | Redis key 格式是缓存读写双方的隐式契约：`<DOC_ROOT>\fms-core\architecture.md` §4 所列消费模块均按旧格式读写；新旧格式并存期间，旧格式读方将全部 miss | **高** | 新代码「双读」（先读新 key，miss 后回落旧 key）+ 过渡期双写；待旧 key TTL 全部过期后再移除旧格式逻辑 |
| R2 | 数据风险 | wiki §7 + fms-diagnose | 不涉及 DB 表结构；Redis 旧格式缓存切换后失效，但 JobOrder 缓存可由 DB 重建（cache-aside），无持久化数据丢失。用 fms-diagnose 的 Redis 检查项确认现存 key 规模与 TTL | 中 | 切换前统计旧 key 数量与 TTL 分布；选择低峰期切换 |
| R3 | 性能风险 | wiki §6 + fms-log | JobOrder 查询位于作业分发热点流程（§6）；切换瞬间缓存集中 miss 会击穿到 DB，形成短时压力尖峰 | 中 | R1 的双读方案可消除集中 miss；上线后用 fms-log 观察 DB 慢查询与接口延迟 |
| R4 | 依赖风险 | wiki §5 + gitnexus-impact-analysis | fms-core 是共享库：gitnexus-impact-analysis（target: JobOrderCacheManager, direction: upstream）列出全部消费方（示例：fms-gateway、fms-jobflow 等多个模块）；任一模块不升级依赖版本，就会出现新旧 key 格式并存 | **高** | 用 gitnexus-impact-analysis 全量确认消费方清单；统一升级共享库版本并制定联动发版计划，禁止长期部分升级 |
| R5 | 部署风险 | wiki §8 | 多个消费模块需协调上线；§8 检查后无需新增配置项。若采用 R1 双读方案，可放宽为分批上线 | 中 | 按 R4 的消费方清单排定上线顺序并写入发布 checklist |
| R6 | 回滚风险 | — | 若无双写，回滚后旧代码读不到新格式 key，等价于再经历一次缓存全量失效 | 中 | 保留双写期内的回滚窗口；回滚步骤：回退版本 → 清理新格式 key |
| R7 | 测试覆盖风险 | gitnexus-debugging | key 生成逻辑的现有单测需同步更新；新旧格式兼容路径（双读/双写）为新增逻辑，尚无测试。用 gitnexus-debugging 确认改动路径上的现有测试范围 | 中 | 补齐兼容用例（命中新 key / 回落旧 key / 双写一致性）后再上线 |
| R8 | 已知问题叠加 | wiki §9 | 检查 §9：示例场景下未见与 Redis key 或该缓存组件相关的已知问题条目 | 低 | — |

**整体结论**（示例）：需先落实 R1/R4 缓解措施（双读双写 + 消费方联动升级计划）再实施。
