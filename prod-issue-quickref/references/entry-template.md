# 问题条目模板与回流规则（条目层）

> 本文管**条目层**——单个生产问题的完整记录；分诊层（速查树）规则见 `SKILL.md` + `node-format.md` / `tree-template.md`。
> 来源：试用期计划 6 月「问题模块框架」遗留项，LOC-104 定稿（2026-07-10）；同日按用户意见从 wiki `_meta` 收编进本 skill 统一管理。
> 三条铁律：**只记真实问题，不为数量硬凑**；**代码 bug 走 T2 升级是正确分流，不硬包装成 AI 闭环或脚本成果**；**凭证/密码不入文档与脚本（命令行/日志/wiki/Agent 对话四不入），输出脱敏**。

## 1. 问题模块的三层结构

| 层 | 位置 | 职责 | 管辖 |
|---|---|---|---|
| **分诊层**（速查树） | `<DOC_ROOT>/生产问题速查.md` | 现象 → 判别 → 一句话根因 + 确认链接；**只定位、不分析** | `SKILL.md` §3 + `node-format.md` + 校验器 |
| **条目层**（问题条目） | `<DOC_ROOT>/issue/<YYYYMMDD>-<短名>.md` | 单个问题的完整记录：现象 / 定位步骤 / 根因 / 解决与升级 / 三类判定 / 可复用性 | 本文 §2 模板 |
| **支撑层**（深度文档） | 子系统目录下 troubleshooting / runbook / `architecture.md §` + 只读脚本 | 完整链路、证据判读表、只读排查 SOP | document-systems / wiki-refine 常规约定 |

三层互链要求：速查叶子**必须**链到条目层或支撑层文档（校验器硬查）；条目**必须**外联 ≥1 个 wiki 节点，并注明速查树入口是否已挂。

存量文件不强制改名（如 `old_project/fabusurfer/issue/OTA命令拉取失败问题.md` 保留原名）；新条目建议带 `YYYYMMDD-` 日期前缀。

## 2. 条目标准模板

新建条目复制以下骨架；**八个正文节保持顺序与标题不变**，没有内容的节写「无」并给一句原因，不留空节。

```markdown
---
title: <一句话问题名>
doc_type: problem
repo: <仓名，如 fabusurfer>
system: <系统，如 meishan-cloud>
module: <子系统/模块，如 port-service>
port: [<meishan|yongzhou|meibei|psa|np|dg|common>]
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [<仓名>, 生产问题, <关键词>]
status: active           # active | expired（生命周期见 §5；expired 由修复落地触发）
expired_at:              # expired 时必填：过期日期
expired_by:              # expired 时必填：触发过期的修复 commit / PR / issue
frequency: normal        # normal | high（高频 = 同类 ≥2 次或 ≥2 人问到）
occurrence_count: 1      # 同类累计发生次数（去重复发时 +1，不新开条目）
reporters_count: 1       # 报告人数
tier_demo:               # T0 | T1 | T2（AI+KB 自助程度，答辩展示口径，见 §3）
result_class: []         # 三类结果，可多选：ai_kb_closed | readonly_script_assist | code_bug_t2
related_scripts: []      # 关联只读脚本相对路径
sources: [<issue 号 / 口述 / 日志等来源>]
---

# <问题名>

## 1. 现象
<用户可观察症状、系统/港口/模块、触发条件、影响范围、时间窗>

## 2. 定位步骤
<按时间/因果顺序：查过的 wiki 节点、code graph 结果、日志/只读工具、关键证据（脱敏）；被否掉的假设也要留一行，防下次重走弯路>

## 3. 根因
<一句话根因链（加粗）+ 必要展开；接口/类/Mapper/表/错误类/日志关键字一律原样、反引号包>

## 4. 解决 / 升级
<现场恢复动作；最终修复（PR/commit/SQL）或升级去向（issue/owner/团队）；依赖侧问题写明归属边界>

## 5. 结果分类
<三类判定 + 理由（见本文 §3）；tier_demo 判级理由一句话>

## 6. 关联 wiki
<≥1 条强制外联：架构 §、troubleshooting、_common、速查树入口>

## 7. 关联代码 / 脚本
<culprit 代码锚点（类#方法、表、配置、commit）+ 关联只读脚本；这些锚点同时是 §5 过期检测的命中依据，写全、写原样>

## 8. 可复用性
<定位路径中哪段可复用（判别序列/分界规则）；是否值得脚本化（须同时满足：重复出现 + 只读安全 + 可参数化，一次性人工判断不硬转）；速查树入口是否已挂>
```

frontmatter 字段与 L1 schema（试用期计划附录 B `doc_type: problem`）对齐：`frequency / occurrence_count / reporters_count / tier_demo / related_scripts` 即附录 B 的 problem 专属字段；`result_class` 为三类结果标记（LOC-104 引入）；L1 管线铺到问题页时再补挂 `code_refs` 等结构锚点，存量优雅降级。

## 3. 三类结果判定

每条问题必须落到下面至少一类（`result_class` 可多选，「定位手段」与「处置去向」是两个维度）：

| 类 | 值 | 含义 | 判定要点 |
|---|---|---|---|
| ① AI+KB 可闭环 | `ai_kb_closed` | AI + 知识库给出根因与可执行方案，提问者/运维照做即闭环，无需开发深入排查 | 对应 `tier_demo: T0`（AI 直接给根因+方案）或 `T1`（AI 给线索/候选，运维自行解决）|
| ② 只读脚本可辅助 | `readonly_script_assist` | 定位路径已沉淀为可复用只读脚本/SOP，下次同类问题可半自动取证 | 脚本化三条件缺一不可：重复出现、只读安全、可参数化；脚本入 `related_scripts` 并在支撑层留判读表 |
| ③ 代码 bug / T2 升级 | `code_bug_t2` | 需改代码/DDL、查深层逻辑，或属依赖方/其他团队问题，转交开发或移交 owner | 对应 `tier_demo: T2`；**必须**链接升级去向（issue/PR/owner）；T2 分流本身是正确结果，「AI+KB 在升级前已完成根因定位」照实记录，即是终态能力证据 |

典型组合：②+③（定位路径可脚本化、修复归开发）、①+②（AI 闭环且脚本化）。禁止：为凑 ①② 把代码 bug 包装成闭环成果。

## 4. 生产问题回流规则

**触发时机**：一次真实生产问题定位收尾时（自然出现原则）。入口有三：本 skill 的 Stop-hook 记录门禁、手动 `/prod-issue-quickref add`、承接/复盘时补登。

**固定动线（四步，顺序执行）**：

1. **速查树**：在 `<DOC_ROOT>/生产问题速查.md` 加或合并分支（按 `SKILL.md` §3 与 `node-format.md`；去重优先，同类分支合并不新开；**必跑校验器**）。
2. **条目**：定位过程有完整证据链 → 按 §2 模板建 `<DOC_ROOT>/issue/` 条目；若只有一句话根因（典型如依赖侧宕机、移交即终态）→ 允许「速查分支即终态」，在周复盘里显式写明，不算欠账。
3. **深度材料**：复杂链路 / 判读表 / 只读脚本 → 放对应子系统目录（troubleshooting / runbook），与条目、速查树互链。
4. **计数**：周五在试用期计划当月月页「术语表与问题模块持续沉淀」机制行记：`问题模块：新增 n（速查分支 x / 条目 y）｜欠账 m（<欠账清单>）`。

**去重与高频**：同类现象再次发生 → 不新开条目，原条目 `occurrence_count +1`、`reporters_count` 更新、正文补差异；累计「同类 ≥2 次或 ≥2 人问到」→ `frequency: high`，高频条目优先补只读脚本与深度文档。

**欠账定义**：已定位的真实问题，四步动线中任一步缺失且未显式声明「速查分支即终态」——例如速查叶子挂着「暂无独立 wiki 文档，建议后续补」。欠账要么下周清掉，要么降级为显式终态声明；欠账清单随周复盘滚动，不允许静默消失。

**数量原则**：新增条目数随真实问题量浮动，是记录口径不是 KPI；连续数周 0 新增 + 0 欠账是正常状态。

## 5. 生命周期与过期（防陈旧条目干扰检索）

问题条目是**事故时点快照**：§3 根因、§7 代码锚点描述的是事发 commit 的逻辑；`code_bug_t2` 类条目在修复落地后必然过期，不处理就会误导后续排障。

**状态机**：`active → expired →（可选）归档移出`。

- **`active`**：根因路径在现网仍可能复发。速查树保留对应分支。
- **`expired`**（修复已落地 / 逻辑已调整）三步，缺一不可：
  1. frontmatter：`status: expired` + `expired_at: <日期>` + `expired_by: <修复 commit/PR/issue>`；
  2. 正文 H1 下加一行：`> ⚠️ 已过期：本条目描述的代码逻辑已随 <commit/PR> 调整，仅作历史复盘素材，勿按本文排障。`；
  3. **同步修剪速查树**：删除或改写对应判别/根因分支——树的契约是「现网当前适用」，过期分支必须出树；改完必跑校验器。
- **归档**：过期条目默认原地保留（复盘/答辩素材仍需要）；若确认干扰日常检索，移入 `<WIKI_BASE>/_archive/<域>/<仓>/issue/`（`_archive` 不在 `.wiki.json` 的 domains 里，不参与域解析与常规检索），并修复指向它的入链。

**触发与分工**：

- **自动**：L2 wiki 自动更新任务（如 LOC-107 的每日 release 分支扫描 + wiki-refine 增量）在变更 diff **直接命中**条目 §7 代码锚点（修复 commit 改了列出的类#方法/表/配置）时 → 直接执行上面三步标 `expired`，并在当日变更总结里列出；命中**模糊**（同文件不同路径、仅间接依赖）→ 不改状态，在条目正文补一行「`<commit>` 触及相关代码，待复核」并列入待过期复核清单交人确认。
- **人工**：修复代码的人 / 承接复盘时可当场手动走同样三步。
- 判定要点：修的是「本条目的根因路径」才过期；无关重构不碰。宁可挂「待复核」也不误杀 active 条目。

## 6. 边界

- 代码审查 / 设计评审中发现的**风险**（未在生产触发、无现象与定位过程，如 PREPDOCKG NPE 风险、HOWO 里程缺口）**不是**生产问题条目——继续走计划仓 `work_record/`；日后在生产实际触发，再按 §4 动线入库并回链原 work_record。
- 本文不改三档 T0/T1/T2 口径本身（定义见试用期计划《试用期管理目标》效果指标节、《阶段二收尾 checklist》真实问题记录口径）。
